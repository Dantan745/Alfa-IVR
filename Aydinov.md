Проблемное поле
Проблемы граждан:
1. Большие комиссии при переводах между банками и через сторонние сервисы.
2. Сильная зависимость между безналичными платежами и интернетом.
3. Риски мошенничества при использовании карт.
4. Неудобство управления личными финансами и отсутствие простой автоматизации платежей. (Например, регулярных закупок еды на неделю производится автоматически)

Проблемы бизнеса:
1. Рост расходов из-за комиссий банков и платёжных систем.
2. Задержки в расчетах с клиентами, то есть трата большого количества времени на это.
3. Отсутствие автоматического контроля оплаты по договорам.
4. Зависимость от иностранных платежных инфраструктур.(создание своей как дополнение к имеющимся)

Проблемы государства:
1. Зависимость национальной платежной системы от зарубежных технологий.
2. Недостаточная прозрачность движения бюджетных средств. Не все операции, которые происходят видны.
3. Высокая доля теневых и наличных расчетов, проблема ограниченности в видимости денежного потока.
4. Ограниченные возможности быстрой и адресной финансовой поддержки. (Например, cоц выплаты или субсидии приходят с задержкой, а не сразу)

Образ продукта
Образ продукта

Название продукта: Alfa Кошелек ЦР - Мобильное и Веб приложение на Android

Тип продукта: Рабочий прототип, доступный через в качестве мобильного приложения(на android) и его Веб версии.

Что будет создано (основные компоненты финальной версии продукта):

Серверная логика и данные (Backend - моя работа(Фариса)):
Простой сервер на Kotlin через JetBrains для Веб и мобильной версии приложения), который, будет обрабатывать взаимодействие пользователя с приложением на системном уровне(переводы средств, обмен цифрового рубля на обычный и обратно, автооплата, история переводов, возможность вернуть средства(через поддержку) в случае если они ушли мошенникам, оплата по QR и другие различные базовые операции(как в обычном приложении банка по типу Сбера или Альфа) реализованные в приложение)
(Будет сделан акцент на разработку функций связанных с обработкой данных с помощью ML)
Помимо того сервер будет связующим звеном при взаимодействие различных систем:
SQL - база данных
Frontend - взаимодействие пользователя с кнопками реализованными в приложении
ML - обработка данных пользователя с помощью ИИ

Также в серверной части частично лежит задача по реализации системы безопасности и защиты данных клиента
---

Рабочий демонстрационный сценарий:
Любой пользователь сможет зайти в приложение через мобильное приложение, произвести транзакцию в виде обмена обычного рубля на цифровой, отправить деньги в виде цифрового рубля другому человеку или настроить автооплату в различных сервисах.

Возможные риски и пути их преодоления
Возможные риски и пути их преодоления (Backend)
1. Технические риски

Риск: Неоптимальный выбор технологий для прототипа.

Пути преодоления:
1. Провести валидацию архитектуры через создание минимального Proof of Concept (PoC) для проверки взаимодействия ключевых компонентов.
2. Применить гибкий подход: использовать Spring Boot (Kotlin) для backend и начать сразу с PostgreSQL.
3. Четко определить и задокументировать контракты API (Swagger/OpenAPI) между всеми модулями (Backend, Frontend, ML) для минимизации рисков интеграции.

Риск: Недостаточный уровень защиты финансового прототипа.

Пути преодоления: Использовать HTTPS, хеширование паролей (bcrypt), защита от SQL-инъекций через ORM. Хранить секретные ключи в переменных окружения.

2. Риски интеграции и командной работы

Риск: Несогласованность между модулями (Frontend, Backend, ML).

Пути преодоления: Первоочередная разработка и документация API (Swagger/OpenAPI) как контракта для всей команды. Четкое определение форматов данных для ML-модуля на этапе проектирования.

Риск: Backend становится узким местом из-за объема работы.

Пути преодоления: Строгая декомпозиция проекта на независимые модули (аутентификация, транзакции, история). Привлечение специалиста по SQL для проектирования схемы данных.

3. Риски качества и надежности

Риск: Нестабильность прототипа из-за недостаточного тестирования.

Пути преодоления: Написание юнит-тестов для ключевой бизнес-логики (транзакции, расчеты). Создание коллекции интеграционных тестов API в Postman. Тестирование на граничных случаях.

Риск: Упрощенная финансовая логика, снижающая правдоподобие.

Пути преодоления: Концентрация на одном-двух сценариях (например, конвертация и перевод) с глубокой проработкой. Обязательное ведение аудит-лога всех операций. Регулярные уточнения бизнес-логики у эксперта-консультанта.
