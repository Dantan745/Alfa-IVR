Название проекта - "Цифровой рубль"

Проект "Цифровой рубль" представляет собой приложение на Android, направленный на моделирование третьей формы денег (цифрового рубля) в Российской Федерации, которая является дополнением к уже существующим формам национальной валюты.

2) Айндинов Фарис, 10И5 - backend-разработка серверной части; 
3) Каплин Степан, 10И5 - frontend-разработка и проведение исследования для создания структуры проекта; 
4) Мифтахов Азар, 10И2 - frontend-разработка и работа с базами данных(SQL);

**[ ПРОБЛЕМНОЕ ПОЛЕ ]**

Зачем нужен проект:
Проект нужен для разработки нового и современного приложение на Android по взаимодействию с цифровым рублем. Основные причины создания проекта:

1. Формирование доступной и защищенной системы платежей на основе цифрового рубля.
2. Создание логичной системы для финансовых операций, с возможностью отслеживать все операции и транзакции.
3. Бесплатные операции для граждан. Для бизнеса вводятся комиссии, которые варьируются в зависимости от величины самой компании.


Какие проблемы данный проект решает:

Проблемы граждан:
1. Отсутствие предсказаний рисков для граждан по различным вкладам.
2. Рекомендательные системы недостаточно четко прогнозируют интересы пользователей (граждан).
3. Неудобство управления личными финансами и отсутствие простой автоматизации платежей. Например, автоматическое выполнение регулярных покупок.

Проблемы бизнеса:
1. Отсутствие аналитики действий клиентов и задержек платежей на основе данных.
2. Сложности с выявлением аномалий. Например, выявление мошенников и нелегальных операций.
3. Отсутствие автоматического контроля оплаты по договорам.

Проблемы государства:
1. Нет автоматизированной поддержки сервиса на основе данных. Например, социальные выплаты или субсидии приходят с задержкой, а не сразу
2. Недостаточная прозрачность потока бюджетных средств. Не все операции прозрачны.


**[ ОБРАЗ ПРОДУКТА ]**

Образ продукта (MVP)

Название продукта: Alfa Кошелек ЦР

Тип продукта: Рабочая демонстрационная версия (MVP) мобильного приложения "Alfa Кошелек ЦР" для операционной системы Android. 

1) Исследование и проектирование интерфейса. Моя работа начнется с анализа современных мобильных банковских приложений. Цель — разработать удобный и интуитивный дизайн, идеально подходящий для сценариев использования цифрового рубля на небольшом экране. Я продумаю расположение ключевых элементов: баланса, истории операций и пошагового конструктора смарт-контрактов. 

2) Создание кликабельного прототипа в Figma. На основе этого анализа я создам в Figma полный макет всех основных экранов приложения. Прототип будет интерактивным, чтобы можно было смоделировать переходы между окнами и основные действия (например, процесс подписания контракта). Это поможет нам всем согласовать логику приложения до начала программирования. 

3) Разработка интерфейса в Android Studio. После утверждения дизайна я приступлю к реализации интерфейса в среде Android Studio с использованием языка Kotlin. Моя задача — превратить статичный макет в живое, отзывчивое приложение: настроить навигацию между экранами, реализовать анимации и обеспечить мгновенную реакцию на действия пользователя. 

4) Организация локального хранилища данных. Чтобы приложение было быстрым и могло частично работать без постоянного подключения к интернету, я настрою локальную базу данных прямо на устройстве. Для этого я буду использовать библиотеку Room, которая упрощает работу с SQLite. В этом локальном хранилище будут кэшироваться данные пользователя и последние операции, что позволит быстро отображать информацию при запуске. 

5) Интеграция с сервером и финальная сборка. На последнем этапе я подключу готовый интерфейс к серверной части (backend), которую разрабатывает мой коллега. Приложение будет обмениваться данными с сервером: отправлять запросы на выполнение операций и получать актуальную информацию, синхронизируя её с локальной базой данных. В итоге мы соберем готовый установочный файл (APK) для демонстрации рабочего прототипа. 

Мой итоговый вклад: Я создам удобное и современное мобильное приложение, в котором пользователи смогут в несколько касаний управлять цифровым рублём: следить за балансом, переводить средства и настраивать автоматические платежи через смарт-контракты. Интерфейс будет простым, а работа — быстрой благодаря грамотному хранению данных прямо на устройстве. 

**[ ВЗАИМОДЕЙСТВИЕ С ДРУГИМИ ЛЮДЬМИ В РАМКАХ ВЫПОЛНЕНИЯ ПРОЕКТА ]** 

Взаимодействие в команде: 

1) С соразработчиком по фронтенду: Мы работаем параллельно над разными частями интерфейса. Чтобы стиль и поведение приложения были одинаковыми, мы регулярно сверяем наш код, договариваемся об общих правилах и используем друг у друга готовые компоненты. Например, я могу использовать созданную им кнопку навигации, а он — мою форму ввода для контракта. 

2) С разработчиком бэкенда: Наше взаимодействие критически важно. Я жду от него готовое и задокументированное API — набор команд, по которым мой интерфейс будет общаться с сервером. Без этого я не могу сделать финальное подключение. В то же время, я предоставляю ему работающие экраны приложения, чтобы он мог протестировать, как сервер реагирует на запросы от реального интерфейса. 

3) С разработчиком по машинному обучению (ML): Мы взаимодействуем на этапе планирования новых функций. Если мы решим добавить аналитику трат, он будет отвечать за алгоритм, а моя задача — встроить результат его работы в интерфейс приложения и красиво отобразить данные для пользователя. 

Работа с консультантами и руководством: 

1) Со специалистом от Альфа-Банка: Его консультации помогают мне правильно понять, как должны работать финансовые операции в приложении. Я уточняю у него детали, чтобы интерфейс и логика данных точно соответствовали реальным бизнес-процессам. 

2) С куратором от Лицея: Я отчитываюсь о прогрессе по своей части, получаю обратную связь по срокам и согласовываю организационные моменты. Его оценка помогает скорректировать приоритеты в моей работе. 

Формат работы: Вся команда синхронизируется на еженедельных созвонах. Дополнительно я провожу короткие обсуждения с бэкенд-разработчиком по техническим вопросам и консультируюсь со специалистом от банка для уточнения требований.

**[ ВОЗМОЖНЫЕ РИСКИ И ПУТИ ИХ ПРЕОДОЛЕНИЯ ]**

1) Риски в разработке: 

   - Недостаток опыта в Kotlin/Android: Может привести к ошибкам и нестабильной работе. 
   - Решение: Активное обучение по курсам, код-ревью с командой, тестирование на разных устройствах. 

   - Сложности с интеграцией API: Нестыковки с сервером из-за изменений или неполной документации. 
   - Решение: Тесное взаимодействие с backend-разработчиком, использование проверенных библиотек (Retrofit). 

   - Проблемы с локальной базой данных (Room): Некорректная работа или уязвимости. 
   - Решение: Следование стандартам безопасности (шифрование, валидация), тестирование сценариев с ошибками. 

2) Риски в проектировании и командной работе: 

   - Неудобный интерфейс: Пользователи могут не понять логику приложения. 
   - Решение: Раннее тестирование прототипов в Figma и гибкая доработка на основе обратной связи. 

   - Несогласованность в команде: Разный стиль кода и конфликты при совместной работе. 
   - Решение: Создание общих правил разработки (code style), использование GitHub для четкого распределения задач. 

3) Организационные риски: 

   - Нехватка времени: Освоение новых технологий может занять больше времени, чем планировалось. 
   - Решение: Реалистичное планирование с запасом, приоритизация базовых функций над улучшениями. 

Общий подход: Поэтапная разработка с фокусом на тестировании каждого компонента. Открытое обсуждение проблем с командой для быстрого нахождения решений.
